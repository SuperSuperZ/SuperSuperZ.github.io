一、Maven是什么？

  • Maven是一个跨平台的项目管理工具。其主要服务于基于Java平台的项目构建、依赖管理和项目信息管理。

  • Maven基本功能：

    • 构建
    • 编译
    • 打包
    • 测试
    • 依赖管理(各种jar包)

  • Maven的特点：

    • 项目设置遵循统一的规则。

    • 任意工程中共享。

    • 依赖管理包括自动更新。

    • 一个庞大且不断增长的库。

    • 可扩展，能够轻松编写 Java 或脚本语言的插件。

    • 只需很少或不需要额外配置即可即时访问新功能。

二、Maven安装和配置

  1. 确定已安装jdk

  2. 下载安装

    • 访问官网下载页面，
      下载系统对应版本的maven安装包，并解压。
      (PS：因为笔者是Windows系统，所以下载的是 apache-maven-3.6.3-bin.zip。)

  3. 配置环境变量

    • 在环境变量中添加变量 MAVEN_HOME，变量值为 C:\Java\apache-maven-3.6.3，
    • 并在已有变量 Path 中追加值 %MAVEN_HOME%\bin。
      (PS：变量值请大家自行更改为自己之前解压的目录路径。)

  4. 检查是否安装成功

    • 运行 CMD 打开命令行，输入mvn --version，若输出版本信息则为安装成功。

  5. 配置本地仓库(可选)

    • 在D盘，新建文件夹 maven-repository
    • 然后在 C:\Java\apache-maven-3.6.3\conf\settings.xml 文件中，将 <localRepository> 节点取消注释。
    • 并更改为 <localRepository>D:\maven-repository</localRepository>。
      (PS：
      配置了本地仓库后，当我们从maven中获取jar包的时候，maven首先会在本地仓库中查找，如果本地仓库有则返回；如果没有则从远程仓库中获取包，并在本地库中保存。
      本地仓库其实起到了一个缓存的作用，它的默认地址是 C:\Users\用户名.m2。)

  7. IDEA中配置maven(可选)

三、Maven构建项目（命令行交互方式）

（一）命令行交互方式

  1. 打开 CMD 命令行，输入 mvn archetype:generate

  2. 命令行将输出一个模板列表，需要我们选择一个模板(Choose archetype)为基础来构建工程

  3. 我们可以键入列表前面的序号或关键字，来缩小模板选择范围或者直接选中模板，这里输入 webapp，筛选出web应用的模板

  4. 键入需要的 模板序号，来选择构建web项目的模板 org.apache.maven.archetypes:maven-archetype-webapp

  5. 选择模板的版本，输入序号回车。若模板只有一个版本，则会跳过这一步

  6. 接着输入项目的坐标：groupId(项目组织ID) 、 artifactId(项目ID)、 version(版本)、 package(包)

    • groupId：项目组织唯一的标识符，默认会对应JAVA的包的结构，是main目录里java的目录结构。
    • artifactId：项目的唯一的标识符，实际对应项目的名称，就是项目根目录的名称。
    • version：项目的版本，“1.0-SNAPSHOT”表示快照1.0版本，“1.0-release”表示发行版1.0

  7. 接着是确认属性配置(Confirm properties configuration)，若信息无误，直接回车即可

  8. 可以看到，当前目录下多了一个以输入的artifactId来命名的文件夹

  9. 到此基本项目构建完成

五、关于pom.xml文件

  1. 刚刚构建的项目文件夹下有一个pom.xml文件。
    　　它是项目的描述文件，描述了项目的maven坐标，依赖关系，开发者需要遵循的规则，缺陷管理系统，组织和licenses，以及其他所有的项目相关因素，是项目级别的配置文件。
    (PS：最基础的pom文件，可以看这里 http://maven.apache.org/pom.html#The_Basics )

  2. web项目的pom.xml结构

    • 特别说明一下依赖节点（<dependency>）下的<scope>
        配置provided相当于compile，但是在打包阶段做了exclude的动作，也就是当我们打包的时候就不会把这个依赖加进去。最常见的是j2ee规范相关的servlet-api和jsp-api等jar包，一般Tomcat会提供，无需在打包到war包中，如果不配置为provided，把这些包打包到工程war包中，在tomcat6以上版本会出现冲突无法正常运行程序（版本不符的情况）。

六、默认约定

  • Maven 提倡使用一个共同的标准目录结构，Maven 使用约定优于配置的原则，大家尽可能的遵守这样的目录结构。
  • 约定优于配置的设定一个是为了简化了配置文件的配置，另外也是为了规范maven工程的目录结构
目录 目的
${basedir} 存放pom.xml和所有的子目录
${basedir}/src/main/java 项目的java源代码
${basedir}/src/main/resources 项目的资源，比如说property文件，springmvc.xml
${basedir}/src/test/java 项目的测试类，比如说Junit代码
${basedir}/src/test/resources 测试用的资源
${basedir}/src/main/webapp/WEB-INF web应用文件目录，web项目的信息，比如存放web.xml、本地图片、jsp视图页面
${basedir}/target 打包输出目录
${basedir}/target/classes 编译输出目录
${basedir}/target/test-classes 测试编译输出目录
TestXXX.java Maven只会自动运行符合该命名规则的测试类
testFunction() Maven只会自动运行测试类中的符合该命名规则测试方法
~/.m2/repository Maven默认的本地仓库目录位置
